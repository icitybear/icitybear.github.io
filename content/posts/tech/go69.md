# 认证和授权
- <font color="red">认证(Authentication)和授权(Authorization)一般是分开的</font>
- 认证：甄别用户是谁
- 授权：判断用户有什么权限（访问控制）
- 授权不管认证，认证也不管授权，二者相互配合
- 所谓的权限，核心逻辑就是："谁（主体）对什么（客体）做什么（操作）"

# 授权模型
ACL、RBAC、ABAC、PBAC 都属于“访问控制模型”或“授权模型”，它们是用于设计和实现系统权限管理的不同方案

## ACL（访问控制列表）
- 为每个资源维护一个 "可访问主体 + 操作" 的列表（如 "资源 A 的访问列表：用户 1（读）、用户 2（读写）"）
- 典型案例：Linux/Unix 文件系统权限：使用 ls -l 命令可以看到 rwx（所有者、所属组、其他人）的权限，贴在文件上的一个“授权名单” 名单里记录了每个用户/用户组对该文件的权限（如：读、写、执行）
- 优缺点：直观易实现，但权限粒度粗（难以批量管理，资源量/用户量增长后列表膨胀）
- <font color="red">ACL是最早也是最基本的一种访问控制机制，每一项资源都配有一个列表，记录哪些用户可以对这项资源执行CRUD中的哪些操作</font>

## RBAC（基于角色的访问控制）
- <font color="red">通过 "角色" 作为中间层，将 "用户-角色-权限" 解耦（用户关联角色，角色关联权限）</font>
- 为什么不直接走用户-权限：假设一个用户有100个权限，后续新用户进入系统需要授权，设计会导致授权流程很复杂。因此可以直接将权限集关联至一个角色上
- RBAC版本：
  - RBAC0（基础版）：用户-角色多对多，角色-权限多对多
  - RBAC1（角色继承）：支持角色间的父子关系（如 "管理员" 继承 "操作员" 的权限）
  - RBAC2（约束控制）：增加角色互斥（如 "出纳" 和 "会计" 不能为同一人）、基数约束等规则
  - RBAC3（综合版）：RBAC1+RBAC2 的结合
- 典型案例：企业内部系统（OA、ERP、CRM）
- <font color="red">目前最主流的权限模型，适合中大型系统</font>

## ABAC（基于属性的权限控制）
- 通过 "主体属性（如用户部门、级别）、资源属性（如文件类型）、环境属性（如登录IP、时间）" 动态计算权限
- 权限 = 主体属性 + 资源属性 + 环境属性 → 操作允许
- 典型案例：复杂合规系统，云资源访问
- 优缺点：灵活性极强（规则可动态调整），但设计复杂（需定义大量属性和规则引擎）
- 适合业务场景多变、安全合规要求高的系统

## PBAC（基于策略的权限控制）
- <font color="red">PBAC采用策略优先的方式进行授权，基于集中化的策略来评估访问请求，策略中可以组合角色、属性和上下文</font>
- 可以理解为RBAC和ABAC的融合升级版，通过统一的策略引擎做决策
- 例如：管理员只能在季度末访问财务报表，而财务分析师全年可访问但仅限工作时间
- 优点：集中化控制，与组织策略对齐，支持多因素决策，适合混合云/多云环境
  - 通过一个中心化的策略决策点来管理所有授权规则，简化了管理，强化了安全，并便于审计
  - 能够根据各种上下文属性 做出更精确的授权决策，超越了仅基于静态角色的限制 
  - 避免了为每个细微的权限组合创建大量独立RBAC角色的需求
- 缺点：需要清晰的策略定义、强治理能力
- 典型案例：Amazon Verified Permissions等云平台权限服务

## 数据权限模型（行/列）
- 通过定义好业务的可见范围，来控制不同租户下的数据查询域
- 例如报表系统中，不同部门的员工所看到的数据范围按照部门字段来筛选区分

## 模型对比
| 维度 | ACL | RBAC | ABAC | PBAC |
|------|------|------|------|------|
| 核心思路 | 基于资源（我是资源主人，我说了算） | 基于角色分配权限 （你是什么角色，就有什么权限）| 基于多维属性动态计算 （根据你现在所有的条件，动态判断）| 基于集中策略统一决策（用集中、可读的策略文件来定义所有规则） |
| 关键实体 | 用户， 资源 | 用户， 角色， 权限| 用户/资源/环境属性 | 策略（包含属性、规则） |
| 灵活性 | 高 | 低，角色固定 | 高，属性可动态组合 | 高，策略可实时调整 |
| 实现复杂度| 低 | 低 | 高 | 中高 |
| 适用场景 | | 角色稳定的中小系统 | 动态多变的复杂系统 | 多云/混合云、合规要求高 |
| 典型问题 | 难以批量管理 | <font color="red">角色爆炸（role explosion）</font> | 属性治理复杂 | 策略设计门槛高 |
| 审计难度 | 难 | 简单 | 较难 | 中等 |

### 如何选择
- 选RBAC：组织角色稳定，需要快速部署和简单审计，IT团队带宽有限
- 选ABAC：环境频繁变化，需要细粒度访问控制满足安全合规要求
- 选PBAC：需要跨多环境/多云平台的集中治理，希望将角色、属性和上下文统一到单一决策框架

- 混合模式
  - <font color="red">实际项目中，最佳方案往往是混合使用多种模型</font>例如：用RBAC处理大部分标准访问需求，叠加ABAC属性来保护敏感数据或执行合规策略

# Casbin
- <font color="red">Casbin：强大且高效的开源访问控制库，支持各种访问控制模型(ACL/RBAC/ABAC/PBAC)，适合做权限控制场景</font>
- Casbin 通过其通用的PERM元模型（策略、效果、请求、匹配器）来抽象访问控制模型
- 各个编程语言都对casbin有支持，比如go语言github.com/casbin/casbin/v2
  - gin中间件(一般自己写)  https://github.com/gin-contrib/authz
  - 策略存储（gorm） https://github.com/apache/casbin-gorm-adapter
  - 角色管理器 https://github.com/apache/casbin/blob/master/rbac/default-role-manager/role_manager.go

- [Casbin支持的模型列表](https://casbin.org/zh/docs/supported-models/)
  - ACL (Access Control List)
  - ACL with superuser 超级用户
  - ACL without users 无用户
  - ACL without resources 无资源
  - RBAC (Role-Based Access Control)
  - RBAC with resource roles 资源角色 用户和资源可以同时具有角色 (或组)。
  - RBAC with domains/tenants 多租户 用户可以为不同的域/租户设置不同的角色集
  - ABAC (Attribute-Based Access Control) 使用json
  - RESTful 支持路径  如/res/*,/res/:id
  - Priority 优先级模型


- [Casbin官方文档](https://casbin.org/zh/docs/overview) 翻译后的 https://www.kancloud.cn/oldlei/casbin/1289450
- <font color="red">[Casbin在线编辑器](https://casbin.org/zh/editor) 练习演示</font>

## 基本概念
casbin有两个部分组成：<font color="red">一个是配置文件(model模型文件)，一个是规则集(policy_rule策略文件)</font>
- model存储，可以从配置文件或者字符串加载
- 一般来说模型文件是放在本地，并且很少更改，而策略文件可以放在很多地方，比如数据库
- 元素为json 判断json里的字段（属性） 
### Request（请求定义）
- 代表一个请求有哪些标准元素（可选 输入据此匹配）
``` text
[request_definition]
r = sub, obj, act
r2 = sub, obj
r3 = sub, dom, obj, act
```
- sub: 访问实体 (Subject)（如用户名）
- obj: 访问资源对象 (Object)（如API路径）
- act: 访问方法 (Action) 比如请求操作（如GET/POST）
- 比如再加个租户/域 dom
这些都是简写命名（可自由命名， 匹配器正确使用就可以）

### Role（角色定义 RBAC）
``` text
[role_definition]
g = _, _
g2 = _, _
g3 = _, _, _
```
- g是role_definition定义的一个角色操作符 不用sub, obj之类的对象名称，仅用"_"作为所需参数的占位符， 两个下划线说明需要两个参数(某些模型不止2个)
  - _, _表示角色继承关系的前项和后项，即前项继承后项角色的权限，简单来说，就是前项拥有后项的全部权限。
- 匹配器配合g函数 m = g(r.sub, p.sub) && g2(r.obj, p.obj) && r.act == p.act, sub有角色 obj也有角色归属。g操作符将r.sub映射成了对应的角色，再将其角色与p.sub比较。因为请求中没有角色的数据，所以必然需要一个映射函数将其转换成对应的角色
- g3(r.sub, p.sub, r.dom) 的第三个参数 dom 并不会对 * 做通配符匹配。g 本质上是角色继承关系匹配，只支持精确匹配，除非你显式开启了角色管理器对通配符或模式匹配（如 keyMatch、regexMatch） g要求请求主体 r.sub 必须在对应 r.dom 域内属于策略中的角色 p.sub。

### Policy和（策略定义）
- Policy代表策略，表示具体的权限定义规则 与策略文件里的对应
``` text
[policy_definition]
p = sub, obj, act
p2 = sub, obj
```
pn为policy type 后续p1 p2等等

- eft 的价值在于你需要同时使用 allow 和 deny 策略的场景。如果只用白名单，不定义 eft 就够了，效果一样。

当 policy_definition 里没有定义 eft 字段时（比如 p = sub, obj, act），Casbin 会默认把所有匹配的策略当作 allow 处理。
当你定义了 eft（p = sub, obj, act, eft），你就可以在 policy.csv 里显式指定 allow 或 deny，实现黑名单机制
``` text
p, zhangsan, web1, read, allow
p, zhangsan, web1, write, deny

e = some(where (p.eft == allow)) — 只要有一条 allow 就通过（白名单模式）
e = !some(where (p.eft == deny)) — 只要没有 deny 就通过（黑名单模式）
e = some(where (p.eft == allow)) && !some(where (p.eft == deny)) — 有 allow 且没有 deny 才通过（deny 优先）
``` 

### 策略文件
``` text
p, zhangsan, web1, read
p, zhangsan, web2, read
p2, wangwu, web1
p2, wangwu, web2
```
分别为策略p, p2定义不同的策略，需要的参数不同

- 在policy.csv文件中(一般存储在数据库里)定义的策略就是policy_rule，和Policy一一对应
- Casbin不生产策略，这个策略我们一般开发时，需要在程序中由用户来进行定义，Casbin只是将用户定义的数据，持久化到数据库进行保存，方便资源访问时进行权限判断

### Matcher（匹配器）
- 有请求，有规则，那么请求是否匹配某个规则，则是matcher进行判断的
``` text
[matchers]
m = r.sub == p.sub && r.obj == p.obj && r.act == p.act
```
- <font color="red">完全匹配时返回true，否则返回false</font>
- 函数g regexMatch 可执行子表达式eval 路径匹配keyMatch1...

### Effect（策略效果）
- 用来判断如果一个请求满足了规则，是否需要同意请求
``` text
[policy_effect]
e = some(where (p.eft == allow))
```
- some表示括号中的表达式个数大于等于1就行
- <font color="red">将request和所有policy比对完之后，所有policy的策略结果(p.eft)为allow的个数>=1，整个请求的策略就是true</font>

- demo
``` text
p, A1, /user, post
g, 张三, A1
g, 李四, A1

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act

1. 角色A1,对/user post 有权限，张三、李四都属于A1。一般情况下，上面的范围定义就对了，只要A1有allow权限，张三李四都可以访问，
2. 但是如果遇到一个特例，我们不想给李四/user post的权限，只给张三，其他A1权限都给
# 策略文件补充
李四，/user,post, deny

[policy_effect]
e = some(where (p.eft == allow)) && !some(where (p.eft == deny))
# 有一个policy的策略结果(p.eft)为allow的个数>=1 且 一个policy的策略结果(p.eft)为deny的个数都不能有（个数<1）
``` 

## 基本使用

### ACL访问控制
``` text
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = r.sub == p.sub && r.obj == p.obj && r.act == p.act || r.sub == "root"
```

policy.csv
``` text
p, zhangsan, web1, read
p, zhangsan, web1, write
p, zhangsan, web2, read
p, zhangsan, web2, write
p, lisi, web1, read
```
单元测试
``` go
func TestACL(t *testing.T) {
	e, err := casbin.NewEnforcer("../../conf/model.conf", "../../conf/policy.csv")
	if err != nil {
		log.Fatal("创建策略引擎失败: ", err)
	}
	tests := [][]interface{}{
		{"zhangsan", "web1", "read"},
		{"zhangsan", "web1", "write"},
		{"zhangsan", "web2", "read"},
		{"zhangsan", "web2", "write"},
		{"zhangsan", "webx", "write"},
		{"lisi", "web1", "read"},
		{"lisi", "web2", "read"},
		{"root", "web1", "read"},
		{"root", "webx", "update"},
	}
	expected := []bool{true, true, true, true, false, true, false, true, true}

	for i := 0; i < len(tests); i++ {
		ok, err := e.Enforce(tests[i]...)
		if err != nil {
			t.Fatalf("请求: %v 对应的期待是是: %t, 发生错误: %s", tests[i], expected[i], err)
		}
		assert.Equal(t, expected[i], ok)
	}
}
```

增加规则
``` go
e.AddPolicy("wangwu", "/users", "read")
e.SavePolicy()
```

删除规则
``` go
e.RemovePolicy("wangwu", "/users", "read")
e.SavePolicy()
```

### ACL超级管理员
- <font color="red">只需要在matcher中加上超级管理员判断即可</font>
``` text
[matchers]
# 只要用户名是root就允许
m = r.sub == p.sub && r.obj == p.obj && r.act == p.act || r.sub == "root"
```

## RBAC访问控制

ACL模型在用户和资源都比较少的情况下没什么问题，但是用户和资源量一大，ACL就会变得异常繁琐。<font color="red">RBAC通过引入角色(role)这个中间层来解决这个问题</font>

- 每个用户都属于一个角色，例如开发者、管理员、运维等
- 每个角色都有其特定的权限，权限的增加和删除都通过角色来进行
- 新增一个用户时，只需要给他指派一个角色，就能拥有该角色的所有权限
- 修改角色的权限时，属于这个角色的用户权限就会相应修改

``` text
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act
```

- <font color="red">RBAC与ACL的不同之处在于多了一个role_definition</font>
- g = _, _ 两个下划线说明需要两个参数
- g(r.sub, p.sub) 是角色操作符，将r.sub映射成对应的角色，再将其角色与p.sub比较

policy.csv
``` text
# 定义角色访问控制权限
p, reader, web1, read
p, reader, web2, read
p, writer, web1, write
p, writer, web2, write
# 定义用户归属角色
g, zhangsan, reader
g, lisi, writer
g, wangwu, reader
g, zhaoliu, reader
g, zhaoliu, writer

```
- p行定义角色的权限
- g行定义用户属于哪些角色
- <font color="red">一个用户可以有多个角色</font>

单元测试
``` go
func TestRBAC(t *testing.T) {
	e, err := casbin.NewEnforcer("../../conf/model2.conf", "../../conf/policy2.csv")
	if err != nil {
		log.Fatal("创建策略引擎失败: ", err)

		{"zhangsan", "web1", "read", true},
		{"zhangsan", "web2", "read", true},
		{"zhangsan", "web1", "write", false},
		{"lisi", "web1", "write", true},
		{"lisi", "web2", "write", true},
		{"lisi", "web1", "read", false},
		{"wangwu", "web1", "read", true},
		{"wangwu", "web1", "read", true},
		{"zhaoliu", "web1", "read", true},
		{"zhaoliu", "web2", "write", true},

	for i := 0; i < len(tests); i++ {
		ok, err := e.Enforce(tests[i][:3]...)
		fmt.Println(tests[i], tests[i][3])
```


## 上下文切换

模型中可以定义多个策略和匹配器，通过上下文切换使用不同的组合

``` text
[request_definition]
r = sub, obj, act
r2 = sub, obj

[policy_definition]
p = sub, obj, act
p2 = sub, obj

[policy_effect]
e = some(where (p.eft == allow))
e2 = some(where (p.eft == allow))

[matchers]
m = r.sub == p.sub && r.obj == p.obj && r.act == p.act
m2 = r2.sub == p2.sub && r2.obj == p2.obj
```

代码中通过 casbin.NewEnforceContext 切换上下文
``` go
// 默认使用 (r, p, e, m)
ok, err := e.Enforce("zhangsan", "web1", "read")

// 使用 (r2, p2, e2, m2)
```

``` go
func RBAC() gin.HandlerFunc {
	return func(context *gin.Context) {
		name, _ := context.Get("token") //获取访问的用户名，context.Request. RequestURI，黄取访问的路径， Request.Method获取

			var msg string

			}
			response.MethodNotAllowed(context, msg, nil)
			return
		} else {
			context.Next()
		}
	}
}
``` 

# 权限模型推演过程
- [casbin权限模型推演](https://mp.weixin.qq.com/s/ZJimz_0M0NSznxZa6DnP2A), 方便理解
1. 从准入开始
  - 权限最开始总是准入，即只有两个选择：允许或者拒绝
  - 最简单的策略就是一个用户列表，在列表中的用户表示允许，反之不允许

2. 加入资源控制
   - 随着项目增长，用户可以操作的项目多起来了，需要在策略里加上用户对应的资源
   - 策略变成：用户, 资源, 操作

3.  将模型和策略解耦
    - <font color="red">策略文件跟匹配模型太耦合，需要将模型提炼出来</font>
    - 定义request_definition（请求定义）和policy_definition（策略定义）
    - 通过matcher灵活控制匹配逻辑，不改动策略文件就可以完成匹配模型的转换

4. 加入策略效果
   - 匹配结果不应只有允许，还需要拒绝(deny)
   - <font color="red">e = some(where (p.eft == allow)) 匹配到任意一个允许就放行</font>
   - <font color="red">e = !some(where (p.eft == deny)) 没有任何一个拒绝就放行（黑名单策略）</font>
