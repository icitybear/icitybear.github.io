# 原理
实现两个不兼容或弱兼容接口之间的适配桥接作用

• 目标 target：是一类含有指定功能的接口

• 使用方 client：需要使用 target 的用户, 使用到它的功能

• 被适配的类 adaptee：和目标类 target 功能类似，但不完全吻合

• 适配器类：adapter：能够将 adaptee 适配转换成 target 的功能类

<font color="red">适配器可担任两个对象间的封装器， 它会接收对于一个对象的调用， 并将其转换为另一个对象可识别的格式和接口。</font>

# 适用场景

- 以前开发的系统存在满足新系统功能需求的类，但其接口同新系统的接口不一致。
- 使用第三方提供的组件，但组件接口定义和自己要求的接口定义不同。


# 常规适配模式
![alt text](image1.png)
• 有一个抽象的 Target interface，具备一种核心功能 Operation

• 有一个 Client 作为使用方，需要使用到 Target 的 Operation 功能

• 分别定义了两种具体类型 ConcreteTypeA 和 ConcreteTypeB，作为 Target 的实现类，都具备 Target 所要求的 Operation 方法

• 有一个 Adaptee 类型，它有 Target 相似但又不完全相同的功能 DifferentOperation

• 接下来，Client 为了能够把 Adaptee 当作 Target 使用，引入了一个适配器类 Adapter

• **在 Adapter 中实现了 Target 所要求的 Operation 方法，<font color="red">同时 Adapter 中引入了 Adaptee 作为成员属性. 在 Adapter 的 Operation 方法中，会调用 Adaptee 的 DifferentOperation 方法，并完成将其适配转换成 Operation 的职责</font>** 适配者类缺少 Operation 由适配器类补全， 适配者类未实现target接口


## 常规案例

- adapter 适配器类 它是一个转换器，通过继承或引用适配者的对象（window），把适配者接口(n)转换成目标接口(m)[用m包装下n]，让客户按目标接口的格式访问适配者。
- <font color="red">adaptee 中缺少了一部分目标 target 的核心能力,需要由适配器 adapter 完成这部分能力的适配补齐.</font>
``` go
// 业务场景描述：
// 通过Lightning接口连接电脑，mac实现Lightning接口，但win的电脑实现的是USB接口，此时需要通过一个适配器，lightning -> USB 如果是type-c的接口呢，增加一个适配器

// Target interface
type computer interface {
	InsertIntoLightningPort() // Operation核心功能
}

// Adpatee 适配者类 被访问和适配的现存组件库中的组件接口。
type windows struct {
}
// Adaptee 类型，它有 Target 相似但又不完全相同的功能
func (w *windows) InsertIntoUSBPort() {
	fmt.Println("windows USB")
}

// Adapter 适配器类 它是一个转换器，通过继承或引用适配者的对象，把适配者接口转换成目标接口，让客户按目标接口的格式访问适配者。
type winAdapter struct {
	winMachine *windows // Adapter引入了 Adaptee 作为成员属性
}

// 适配者类缺少 InsertIntoLightningPort 由适配器类winAdapter补全
// tag： 适配者类可能未实现target的interface接口
func (w *winAdapter) InsertIntoLightningPort() {
	fmt.Println("Adapter converts Lightning signal to USB.")
	w.winMachine.InsertIntoUSBPort() // 要使用USB 但是要包装成Lightning  调用 Adaptee 的 DifferentOperation 方法，并完成将其适配转换成 Operation 的职责
}

// implement target struct
type mac struct {
}
func (m *mac) InsertIntoLightningPort() {
	fmt.Println("mac Lightning")
}

// 使用方client
type client struct {
}

// client 让客户按目标接口的格式访问适配者  target接口类型为参数
func (c *client) insertLightningConnectorIntoComputer(com computer) {
	fmt.Println("Client inserts Lightning connector into computer.")
	com.InsertIntoLightningPort()
}

func TestX(t *testing.T) {
	client := &client{}

	// 无需适配已经实现的了
	mac := &mac{}
	client.insertLightningConnectorIntoComputer(mac)
	// Client inserts Lightning connector into computer.
	// mac Lightning

	// 适配器进行了转换
	windowsMachineAdapter := &winAdapter{
		winMachine: &windows{}, // 传入Adaptee 作为成员属性
	}
	client.insertLightningConnectorIntoComputer(windowsMachineAdapter)
	// Client inserts Lightning connector into computer.
	// Adapter converts Lightning signal to USB.
	// windows USB
}

```

# interface模式
- 定义具体类型时，不需要显式声明对 interface 的 implement 操作，只需要实现了 interface 的所有方法，就自动会被编译器识别为 interface 的一种实现.这种特性使得 interface 本身也具备了适配器的功能
- <font color="red">一个结构体struct实现所有方法， 不同的适配器拥有的方法自定义聚合成interface</font>

## 隐式继承
``` go
type MyInterface interface {
    MethodA()
    MethodB()
}

type MyClass struct{}
func NewMyClass() *MyClass {
    return &MyClass{}
}
// MyClass 实现了 MyInterface 声明的所有方法, 所以隐式继承了
func (m *MyClass) MethodA() {}
func (m *MyClass) MethodB() {}

func Test_implement(t *testing.T) {
    // 获取 myClass 的类型
    myClassTyp := reflect.TypeOf(NewMyClass())
    // 获取 myInterface 的类型
    myInterTyp := reflect.TypeOf((*MyInterface)(nil)).Elem()
    // 判断是否具有实现关系
    t.Log(myClassTyp.Implements(myInterTyp)) // 打印为true
	
	// 后续使用函数为MyInterface接口类型时，参数随意class  只要实现了该接口就行，这也是一种适配器模式
}
```
## <font color="red">interface 扮演了适配器的角色</font>
- <font color="red">adaptee 不仅具备 target 的全部能力，还有其他能力，实现了target接口</font>

1. interface 建立接口规范, 抽象出了同一类型下多种角色的共性，将其声明成一个接口规范的形式，最终所有实现类都需要实现 interface 的所有方法
2. 通过 interface 隐藏实现细节，在模块间进行类的传输时，为了保护具体的实现类隐藏其中的实现细节，转而使用抽象 interface 的形式进行传递. 同时这种基于 interface 进行传递参数的方式，也基于了使用方一定的灵活度，<font color="red">可以通过注入 interface 不同实现类的方式，赋予方法更高的灵活度. </font>
  - interface 的定义应该由模块的使用方而非实现方来进行定义. 只有这样，使用方才能根据自己的视角，对模块进行最合适或者说最贴合自己使用需求的抽象定义.

- <font color="red">target从interface直接改为struct，实现全部方法  CourseService 的使用方，定义各种定制的interface</font>
![alt text](image2.png)
- 实现方不再进行 interface 的声明，而是将 CourseService 改为一个具体的实现类型struct，<font color="red">将定义 interface 的职责转交给 CourseService 的使用方 </font>
- 使用方在使用 CourseService 时，<font color="red">根据使用到的 CourseService 的功能范围，对其身份进行抽象和定义</font>，比如在使用 CourseService 中编程课程有关的方法时，使用方可以定义出一个 CSCourseProxy 的 interface，然后在 interface 中定义好有关编程课程的几个方法，其他无关的方法不再声明，起到屏蔽无关职责的效果 

``` go
// target从interface直接改为struct，实现全部方法  CourseService 的使用方，定义各种定制的interface 
type CourseService struct {
}

func NewCourseService() *CourseService {
    return &CourseService{}
}

func (c *CourseService) LearnGolang() {
    fmt.Println("learn go...")
}
// ...余下省略

// 作为适配器 约束 其他无关的方法不再声明，起到屏蔽
type CSCourseProxy interface {
    LearnGolang()
    LearnJAVA()
    LearnC()
}

type MusicCourseProxy interface {
    LearnPiano()
    LearnHarmonica()
    LearnSki()
}

type GolangCourseProxy interface{
    LearnGolang()
}

func Test_golangCourseProxy(t *testing.T) {
	// proxy 是GolangCourseProxy类型（实现了GolangCourseProxy接口），这样一看该变量的职责就很清楚
    var proxy GolangCourseProxy = NewCourseService() // 使用方的实例
    proxy.LearnGolang() // 且该类型调用的只有这个方法，没暴露CourseService类型其他方法
}
```

## <font color="red">2种模式区别</font>
- 常规模式：指的是被适配对象 adaptee 中缺少了一部分目标 target 的核心能力，需要由适配器 adapter 完成这部分能力的适配补齐.
- interface模式: adaptee 不仅具备 target 的全部能力，还聚合了一部分 target 本身不关心的能力. 因此倘若我们直接把 adaptee 当作 target 使用，这部分不相干能力也会被暴露出来，最终对 target 的使用方造成困惑, 通过对 interface 的合适定义使其充当适配器的角色


# <font color="red">使用函数作为适配器</font> 推荐使用
- **<font color="red">type xxx func 定义函数类型（本质值类型，所以可以绑定方法）, 通过适配器将函数转换为接口, 尤其是当你只需要实现接口的一个功能时</font>**
- <font color="red">定义函数类型作为适配器, 注入自定义函数</font> FT定义的函数形参和返回值与自定义函数fVar一致就行

## 通过适配器将函数（func）转换为接口（type）优点
1. **灵活性和可扩展性**：因为你可以传递任何满足接口要求的函数，所以这种方法提供了更多的灵活性
2. **简化实现,减少样板代码**：在某些情况下，定义一个新的结构体类型来实现一个接口可能会很繁琐，尤其是当你只需要实现接口的一个功能时。<font color="red">使用函数作为适配器可以消除这种重复的样板代码。</font>只需提供一个函数来实现接口，而不需要定义一个新的类型.

## 例子
1. <font color="red">类型实现自己的成员方法(可以去实现各种接口，对应的函数就能自定义注入这个方法了)  注意这里不使用 *FT</font>
2. **<font color="red">FT就相当于接口的适配器。IFC与fVar关系 用一个接口(IFC)通过适配器FT(FT类型实现接口IFC （适配器FT实现接口就行）) 去接收一个函数闭包(fVar) 不关心函数(fVar)的实现。(也就实现了灵活注入)</font>** 
   - **<font color="red">FT定义的函数形参和返回值与fVar一致就行</font>** 
3. <font color="red"调用方client: 只依赖接口(作为参数)，不依赖具体实现（依赖倒置原则）。</font>只需要定义IFC接口，无需关心实现，然后自定义函数用适配器FT包装（该适配器函数实现了该接口）
   - <font color="red">用一个接口去接收一个函数，接口规定了函数类型，但是不关心函数的实现。</font>

``` go
// tag: 1. IFC此时就是target
// tag: IFC与fVar关系 用一个接口(IFC)通过适配器FT(FT类型实现接口IFC （适配器FT实现接口就行）) 去接收一个函数闭包(fVar) 不关心函数(fVar)的实现。
type IFC interface { // 参照http.Handler接口
	HelloOpt(arg int)
}

// type定义了个新类型（结构体）, 还有函数作为适配器的写法 尤其是当你只需要实现接口的一个功能时

// tag: 2.函数变量或者匿名函数 此时就是适配者（只是这个适配者只需要适配一个函数） 要注入的函数 自定义的实现（是FT类型）
var fVar = func(arg int) error {
	fmt.Println("fVar use arg:", arg)
	return nil
}

// tag: 3.FT此时就是对应的适配器类 本质值类型，所以可以绑定方法 不是结构体struct 所以创建实例的时候不能用{}
// tag: 通过FT(要包装的具体函数变量fVar或匿名函数xxx)调用 创建FT类型实例, 然后调用这个适配器的方法
type FT func(arg int) error // 可以配置自定义的形参和返回值

// tag: 类型实现自己的成员方法 适配器方法 (可以去实现各种接口，对应的自定义注函数就能入这个方法了)
// tag: 注意这里不使用 *FT,因为调用的时候不需要生成实例再调用
// tag: 使用函数HelloOpt包装了要执行的的闭包f（类型FT） 把函数f注入了接口IFC的方法HelloOpt的流程  f => FT需要是个函数类型
func (f FT) HelloOpt(arg int) {
	// tag: 参数arg就是外部变量
	fmt.Println("FT HelloOpt before", arg)
	f(arg) // 这里就是调用闭包函数 （闭包就是适配者函数）
}

// tag: 4.此时就是client 只依赖接口，不依赖具体实现。依赖倒置原则。
func ClientAdaptee(ifc IFC) {
	ifc.HelloOpt(10)
}

func TestXxx(t *testing.T) {
	fmt.Println(&fVar) // 0x102845868 变量的地址
	fVar(3)            // fVar 3 直接调用函数类型的变量

	// fVar 是一个函数值（函数变量），没有调用，因此会打印函数的类型而不是函数的值
	fmt.Printf("fVar p:%p, t:%T\n", fVar, fVar) // fVar p:0x102ba8e50, t:func(int) error
	// tag: 底层一样，通过FT包装 函数类型FT转换包装 直接指向的函数内存地址还是一样的
	fmt.Printf("fVFT(fVar) p:%v, t:%T\n", FT(fVar), FT(fVar)) // fVFT(fVar) p:0x102ba8e50, t:adaptee_test.FT

	// FT的实例此时也是函数变量 FT函数类型创建实例, 传递一个函数类型进来（此时fVar就是注入的闭包函数)
	_ = FT(fVar)(3) // fVar use arg: 3 这里的3是fVar闭包要的外部变量的传参
	//错误写法: FT(fVar(3)) // FT是函数类型

	// 调用FT类型的成员方法  tag: 绑定HelloOpt直接绑类型不绑指针（函数类型不能再使用指针）
	FT(fVar).HelloOpt(4) // FT HelloOpt before 4 和 fVar use arg: 4
	// (*FT).HelloOpt(&FT(fVar)(3), 4) // 	结构体可以使用& 而函数类型不行 (*Student).Hi(&s)

	// tag: 使用了适配器FT 适配者是自定义方法fVar

	// 只依赖接口，不依赖具体实现。依赖倒置原则。ClientAdaptee只需要定义IFC接口，无需关心实现，然后自定义函数用适配器FT包装（该适配器函数实现了该接口）
	ClientAdaptee(FT(fVar)) // FT HelloOpt before 10 和 fVar use arg: 10
	// x := FT(fVar) // var x IFC
	// ClientAdaptee(x)
}
```

## 常见的http标准库 func -> Type
- 标准库 net.http包 比如Handle调用 使用了适配器HandlerFunc
``` go

type Handler interface {
	ServeHTTP(ResponseWriter, *Request)
}

type HandlerFunc func(ResponseWriter, *Request) // HandlerFunc类型就类似FT, 实现了http.Handler接口（http.Handle的参数要求），HandlerFunc就是适配器了

// 适配器HandlerFunc实现了ServeHTTP该方法 就是实现了http.Handler接口（该接口只定义了这个方法）
// HandlerFunc实现了ServeHTTP该方法 就是实现了http.Handler接口
func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
	f(w, r) // 闭包f
}

// 调用者 参数http.Handler接口（不用关心实现）
http.Handle(pattern string, handler http.Handler)
// 实现就可以通过函数注册的方式进入到xxx的流程里面。同时还可以用于实现观察者等模式。
func Boy(w http.ResponseWriter, r *http.Request) {
	fmt.Println("工")
	w.Write([]byte("boy"))
}

// 例子: http.Handle("/", http.HandlerFunc(Boy)) 真正要执行的函数是boy 注入的闭包
``` 

比如net的http包，http.Handle只依赖接口，不依赖具体实现, 依赖倒置原则。<font color="red">用一个接口去接收一个函数，接口规定了函数类型，但是不关心函数的实现。</font>
